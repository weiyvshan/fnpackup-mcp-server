# build_fpk 修复说明

## 问题描述

**影响版本**: fnpackup-mcp-server@1.0.1、1.0.2

`build_fpk` 错误地本地调用 fnpack CLI，而不是调用 fnpackup 服务 API。

### 实际行为

`dist/tools/builder.js` 在 `buildFpk` 里直接 `spawn("fnpack build ...")`，把本地 fnpack 当作打包后端。

### 问题分析

1. **职责错位**: fnpackup 本体是一个带 Web UI 的服务，MCP server 应该通过 HTTP 调用其接口（如 http://localhost:1069/...），而不是绕过服务层直接调 CLI
2. **产物路径不可控**: fnpack 的行为在不同系统/版本可能不同；代码把 `findGeneratedFpk` 写成只在应用目录或 /tmp/fnpack.*/ 搜索，覆盖不到真实产物位置
3. **Hermes 环境下无状态服务**: 本代理 MCP 的 stdio 进程不能保证每次 build 后产物被正确回收，导致"打包完成但未找到 .fpk 文件"
4. **用户需要额外补齐 config/privilege、config/resource、ICON.PNG**: 才能让本地 fnpack 过验证，但这属于 fnpackup 服务层应处理的应用完整性检查，不应甩给 MCP 使用者

## 修复方案

### 核心变更

修改 `src/tools/builder.ts`，将 `buildFpk` 方法从直接调用 fnpack CLI 改为通过 HTTP 调用 fnpackup 服务 API。

#### 主要修改点

1. **移除 fnpack CLI 调用逻辑**
   - 删除 `findFnpack()` 方法
   - 删除 `executeCommand()` 方法
   - 删除 `findGeneratedFpk()` 方法
   - 删除 `findFpkInTempDir()` 方法
   - 移除 `child_process` 和 `util` 依赖

2. **新增 HTTP API 调用**
   - 新增 `callFnpackupService()` 方法
   - 向 `http://localhost:1069/project/pack` 发送 POST 请求
   - 解析响应并返回生成的 .fpk 文件路径

3. **新增环境变量配置**
   - 新增 `FNPACKUP_SERVICE_URL` 环境变量
   - 默认值: `http://localhost:1069`
   - 允许用户自定义 fnpackup 服务地址

4. **改进错误处理**
   - 当无法连接到 fnpackup 服务时，提供清晰的错误信息
   - 提示用户检查 fnpackup 服务是否已启动

### 技术细节

#### 修改前

```typescript
// 查找 fnpack 可执行文件
const fnpackPath = this.findFnpack();

// 执行打包
const result = await this.executeCommand(fnpackPath, args, {
  cwd: this.appManager.projectsDir || path.dirname(appPath),
});

// 查找生成的 fpk 文件
const generatedFpk = await this.findGeneratedFpk(appPath);
```

#### 修改后

```typescript
// 调用 fnpackup 服务 API 进行打包
const result = await this.callFnpackupService(appName);

// 检查服务是否可用
if (!result.success) {
  throw new Error(result.error || "打包失败");
}
```

### API 调用说明

**Endpoint**: `POST /project/pack`

**请求参数**:
- `name` (query): 应用名称
- `uspace` (query, optional): 用户空间标志
- `platform` (query, optional): 平台标识
- `server` (query, optional): 服务器路径，默认 "app/server"

**响应格式**:
```json
[
  {
    "FileName": "appname-1.0.0-x86_64.fpk",
    "Msg": "Packing successfully"
  }
]
```

**响应说明**:
- `FileName`: 生成的 .fpk 文件名
- `Msg`: 打包消息，成功时包含 "Packing successfully"
- .fpk 文件会被服务自动生成到应用目录 `./projects/{appname}/`

## 配置更新

### 新增环境变量

- **FNPACKUP_SERVICE_URL**: fnpackup 服务的 API 地址（默认: `http://localhost:1069`）

### Claude Code 配置示例

```json
{
  "mcpServers": {
    "fnpackup": {
      "command": "fnpackup-mcp-server",
      "env": {
        "PROJECTS_DIR": "/path/to/fnpackup/projects",
        "FNPACKUP_SERVICE_URL": "http://localhost:1069"
      }
    }
  }
}
```

**注意**: 请根据你的操作系统替换 PROJECTS_DIR：
- **Windows**: `C:\\path\\to\\fnpackup\\projects`
- **macOS/Linux**: `/path/to/fnpackup/projects`

## 依赖变更

### 移除依赖

- 不再依赖本地 fnpack CLI 工具
- 不再需要 `child_process` 模块
- 不再需要 `util.promisify`

### 新增依赖

- 使用 Node.js 18+ 内置的 `fetch` API（无需额外安装）

## 前置要求

1. **必须启动 fnpackup 服务**: MCP server 现在依赖 fnpackup Web 服务来处理打包操作
   ```bash
   cd fnpackup
   dotnet run
   ```

2. **Node.js 18+**: 确保使用 Node.js 18 或更高版本以支持原生 fetch API

## 测试建议

1. **正常情况**: 确保 fnpackup 服务已启动，尝试打包应用，验证 .fpk 文件正确生成
2. **服务未启动**: 停止 fnpackup 服务，尝试打包应用，验证错误信息清晰
3. **服务错误**: 模拟 fnpackup 服务返回错误，验证错误消息正确传递

## 优势

1. **职责清晰**: MCP server 作为 fnpackup 服务的 Agent 接口，符合架构设计
2. **产物路径可控**: 由 fnpackup 服务管理 .fpk 文件的生成和位置，避免了路径搜索的不确定性
3. **服务层完整性检查**: fnpackup 服务负责处理 config/privilege、config/resource、ICON.PNG 等完整性检查
4. **更好的错误处理**: 提供清晰的连接失败信息，便于用户诊断问题
5. **Hermes 环境兼容**: 通过 HTTP 调用服务，不依赖本地 fnpack 进程状态
